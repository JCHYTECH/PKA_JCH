Architecture

Comment ça fonctionne

Bonne nouvelle : le kernel Linux moderne charge automatiquement le driver cdc_ether sur la ZTE MF833U1. Le modem apparaît comme une interface réseau Ethernet (eth1) — pas besoin de PPP, wvdial, ou chat script. La documentation en ligne est souvent obsolète sur ce point.

iPhone (hotspot) ──┐
                  ├──▶ Raspberry Pi 5 ──▶ BirdNET-Go / services
ZTE MF833U1 (LTE) ─┘
  │
  └── eth1 (cdc_ether) — IP DHCP 192.168.0.x
        Gateway modem : 192.168.0.1
        Internet : TCP/HTTP ✅ — ICMP/ping ❌ (bloqué Proximus)

Le modem ZTE gère lui-même la connexion cellulaire et fait du NAT. Il fournit une IP locale au RPi via DHCP. Le RPi voit internet via eth1 exactement comme avec un câble Ethernet.

!
Avant de commencer

Prérequis — à faire une seule fois

1. Activer la carte SIM Proximus Pay&Go

Piège : sans activation, le modem semble fonctionner (DHCP, IP locale, 1 ping sur 20 passe) mais toutes les requêtes HTTP sont interceptées par le portail captif Proximus. L'activation est obligatoire avant tout usage data.

  1. Insérer la SIM dans un téléphone (pas dans la ZTE)
  2. Appeler le 1911 depuis cette carte SIM — activation automatique, message en 3 langues, ~30 secondes
  3. Après confirmation, retirer la SIM du téléphone

2. Désactiver le PIN de la SIM

Piège : à chaque redémarrage du modem ZTE avec une SIM protégée par PIN, l'interface web demande le PIN manuellement. En déploiement terrain autonome, le modem reste bloqué sans connexion data.

  1. SIM dans l'iPhone
  2. Réglages → Téléphone → PIN de la carte SIM → Désactiver le PIN
  3. Entrer le PIN pour confirmer (PIN Proximus par défaut imprimé sur la pochette SIM)
  4. Retirer la SIM — elle démarrera désormais sans PIN dans la ZTE

01
Installation

Setup RPi

Ordre de démarrage — important

  1. Démarrer le RPi sans la ZTE branchée — attendre le boot complet (~45s)
  2. Brancher la ZTE sur le port USB noir
  3. Attendre 10 secondes

Piège confirmé : brancher la ZTE pendant le boot du RPi peut provoquer un crash (pic de courant USB). Le RPi redémarre ou freeze, perd le réseau Wi-Fi. Toujours brancher la ZTE après le boot.

Installer les paquets nécessaires (une seule fois)

sudo apt update && sudo apt install -y usb-modeswitch usb-modeswitch-data ppp wvdial

Sur Debian Trixie, usb-modeswitch et ppp sont souvent déjà présents. Seul wvdial peut manquer — mais il ne sera pas utilisé (cdc_ether rend PPP inutile).

Vérifier la détection de la ZTE

lsusb | grep -i zte
# Attendu : Bus 003 Device 003: ID 19d2:1706 ZTE WCDMA ... DEMO Mobile Boardband
# Le mode DEMO est normal — le kernel charge cdc_ether automatiquement
ip addr show eth1
# Attendu : inet 192.168.0.xxx/24 — IP DHCP du modem
ip route show
# Attendu : default via 192.168.0.1 dev eth1 (route vers internet via modem)
#           default via 192.168.1.1 dev wlan0 (route Wi-Fi maison)
02
Interface web ZTE

Configurer et connecter le modem

Accéder à l'interface ZTE depuis le Mac

Le modem a une interface web sur 192.168.0.1 — accessible uniquement depuis le RPi. Créer un tunnel SSH depuis le Mac :

# [Mac] — ouvrir dans un terminal séparé, laisser tourner
ssh -L 8181:192.168.0.1:80 jchavaux@192.168.1.48 -N

Puis ouvrir http://localhost:8181 dans le navigateur.

Entrer le PIN SIM (si non désactivé)

Au premier accès, l'interface demande le PIN. L'entrer une fois. Idéalement, désactiver le PIN en amont (voir Prérequis) pour éviter cette étape à chaque redémarrage terrain.

Vérifier l'APN

Navigation : Settings → APN. L'interface ZTE détecte Proximus automatiquement :

Forcer la connexion data via API (si nécessaire)

Si la connexion n'est pas établie automatiquement, depuis le RPi :

# Déconnecter
curl -s -X POST -H 'Referer: http://192.168.0.1/index.html' \
  -d 'isTest=false&goformId=DISCONNECT_NETWORK' \
  'http://192.168.0.1/goform/goform_set_cmd_process'

sleep 4

# Reconnecter
curl -s -X POST -H 'Referer: http://192.168.0.1/index.html' \
  -d 'isTest=false&goformId=CONNECT_NETWORK' \
  'http://192.168.0.1/goform/goform_set_cmd_process'

Tests

Vérifier la connectivité

Ne pas utiliser ping pour tester. Proximus LTE bloque l'ICMP. Un ping qui échoue ne signifie pas que la connexion est morte — utiliser exclusivement curl HTTP.

Test de connectivité HTTP (référence)

# Test rapide — doit retourner "success" ou HTTP 200
curl -s --interface eth1 --max-time 10 http://detectportal.firefox.com/success.txt

# Test avec métriques
curl -s --interface eth1 --max-time 15 \
  -o /dev/null \
  -w 'HTTP %{http_code} — %{time_total}s — %{speed_download} B/s\n' \
  http://detectportal.firefox.com/success.txt

LTE

Qualité de signal

> −85 Excellent
−85 à −95 Bon
−95 à −105 Acceptable
−105 à −115 Limite
< −115 Inutilisable

Valeur mesurée en intérieur lors du setup : −107 à −110 dBm. En extérieur ou près d'une fenêtre, attendre −95 à −99 dBm minimum pour un usage terrain stable.

Vérifier le signal depuis le RPi

# Via l'interface web ZTE — valeur en bas à droite "Signal Strength"
# Accessible via le tunnel SSH : http://localhost:8181

# Ou via nmcli
nmcli device show eth1 | grep -i signal
